## 一、 课程核心逻辑理解：硬科技的“生存法则”

我们小组一致认为，本课程的核心不在于“写代码”或“讲故事”，而在于解决**科技成果转化中的“死亡之谷”问题**。

* **技术深度（The "Hard" Part）：** 我们理解的硬科技必须具备物理、工程或算法层面的极高壁垒（如 $10^{-9}$ 级的精度或突破性的能源转化率），而非简单的商业模式套利。
* **创业宽度（The "Venture" Part）：** 创新不仅仅是发明，而是“发明 + 商业化”。我们需要在技术可行性、市场需求、财务模型和法律合规（IP）之间找到交集。
* **核心目标：** 将一个实验室级的原型（Prototype）推向具备市场竞争力的产品（Product）。

## 二、 课程节奏与团队作战图

我们将课程周期划分为四个关键战役，并明确了团队分工：

| 阶段 | 关键任务 (Milestones) | 团队协作重点 |
| :--- | :--- | :--- |
| **P1: 赛道锚定** | 专利检索、技术查新、痛点挖掘 | 技术负责人主导技术可行性评估 |
| **P2: 逻辑闭环** | 最小可行性产品 (MVP) 定义、供应链初探 | 市场负责人进行潜在客户访谈 |
| **P3: 商业建模** | 成本核算、融资计划、风险预估 | 财务负责人构建收益模型与风险矩阵 |
| **P4: 终极交付** | 商业计划书 (BP) 撰写、路演 (Pitching) | 全员参与，打磨极具说服力的演示内容 |

---

## 三、 评价方式与交付质量准则

为了确保项目在期末获得优秀评价，我们小组确立了以下内部交付标准：

1.  **数据驱动（Data-Driven）：** 所有的市场预测必须有可靠的数据来源，所有的技术参数必须经过仿真或实验验证。
2.  **版本受控（Version Control）：** 利用本仓库进行严格的版本管理，确保每一版交付物都有迹可循，体现迭代过程。
3.  **真实挑战：** 我们将针对现实中的硬科技瓶颈（如国产替代、能源转型等）进行选题，确保项目的社会价值与商业潜力。
4.  **跨学科协同：** 确保技术文档（Tech Specs）与商业计划（Business Plan）不脱节，实现真正的“技商融合”。

---

## 四、 团队文化与协作共识

* **快速失败，尽早迭代：** 面对硬科技的高不确定性，我们鼓励在早期发现技术漏洞，而非掩盖问题。
* **GitHub 原生流：** 坚持使用 Markdown 记录会议纪要、调研报告及技术方案，保持文档的清晰与易读。
* **目标导向：** 每一位成员的产出最终都要指向“解决用户痛点”和“提升技术壁垒”。
