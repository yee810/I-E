# Jobro 前端项目分析

## 1. 分析范围与结论

本分析基于 `Jobro/README.md`、`src/` 下全部前端源码、构建配置和依赖声明完成，未包含任何后端服务代码。

一句话判断：

这是一个完成度较高的产品原型前端，而不是一个已经打通真实业务链路的正式生产前端。

更准确一点说，它的强项在于：

- 视觉风格统一，核心页面都已经有高保真界面
- 用户路径比较完整，已经覆盖注册、资料录入、偏好设置、聊天、职位列表、个人资料、设置
- 交互方向清晰，明显在验证“AI 职业助手 / 求职陪跑”这个产品概念

它的短板也很明确：

- 业务数据几乎全部是前端硬编码
- AI 能力只接到了聊天文本生成，职位匹配本身仍是 mock
- 没有真正的状态持久化、权限控制、表单校验、上传处理、后端接口联动
- 存在几处结构性问题和至少一处高优先级运行时风险

我的整体判断是：

这个项目适合作为“接手后继续演进的产品原型底座”，不适合作为“直接上线的完整前端”。

## 2. 项目定位判断

从以下信息可以推断，这个项目大概率是从 Google AI Studio 导出的应用原型：

- `README.md` 只有最基础的本地运行说明，并直接指向 AI Studio 应用链接
- `.env.example` 提到 `GEMINI_API_KEY` 会由 AI Studio secrets 注入
- `package.json` 项目名仍然是 `react-example`

这意味着它更像是：

- 一个用于演示产品概念的原型仓库
- 或者一个从生成式工具导出的前端初稿

而不是已经经过工程化整理的团队正式仓库。

## 3. 技术栈与工程现状

当前前端实际采用的技术栈：

- React 19
- TypeScript
- Vite
- React Router
- Tailwind CSS v4
- `motion` 做动效
- `lucide-react` 做图标
- `@google/genai` 直接在前端调用 Gemini

工程上有两个比较明显的特点：

### 3.1 实际是纯前端单页应用

入口非常简单，`src/App.tsx` 里直接用 `BrowserRouter` 做路由，整个应用就是一个标准 SPA。

### 3.2 依赖声明比实际落地更“重”

`package.json` 里声明了这些依赖：

- `express`
- `better-sqlite3`
- `dotenv`
- `@types/express`

但在当前仓库源码中没有看到对应后端实现或服务入口，说明：

- 要么后端计划过但没有落地
- 要么原本有服务端设计，但这次交接只保留了前端部分

所以现在这个仓库的真实状态，不是“全栈已成型”，而是“前端原型明显领先于后端实现”。

## 4. 页面与用户流程

当前路由定义在 `src/App.tsx`，主要页面如下：

| 路由 | 页面 | 作用 |
| --- | --- | --- |
| `/` | RegistrationScreen | 学校邮箱验证 |
| `/profile-input` | ProfileInputScreen | 上传简历、补充外链 |
| `/preferences` | PreferencesScreen | 录入求职偏好 |
| `/chat` | ChatScreen | AI 对话与职位推荐 |
| `/profile-confirmation` | ProfileConfirmationScreen | 个人资料确认与编辑 |
| `/dashboard` | DashboardScreen | 已投递 / AI 推荐职位列表 |
| `/settings` | SettingsScreen | 设置页 |

### 4.1 设计出来的产品主路径

从交互设计上看，产品想表达的流程是：

1. 用户先完成邮箱验证
2. 上传简历和链接资料
3. 录入求职偏好
4. 进入 AI 对话
5. AI 总结画像
6. 用户确认画像
7. 查看推荐岗位并做投递管理

这是一个很合理的求职助手 onboarding 流程。

### 4.2 当前代码中的真实路径

代码里的真实情况比设计稿要“短路”一些：

- `PreferencesScreen` 点击下一步后直接 `navigate("/chat")`
- `ChatScreen` 默认第一条消息就是职位推荐卡片
- `ChatScreen` 里虽然存在“onboarding 完成后显示 Profile Summary”的逻辑，但它依赖 URL 参数 `?onboarding=complete`
- 当前项目中没有任何地方会导航到 `/chat?onboarding=complete`

这意味着：

- 画像确认卡片在正常流程里实际上不会触发
- 用户会直接从偏好页跳到职位推荐
- “对话总结画像 -> 用户确认画像 -> 再推荐职位”的链路目前没有真正闭环

这是当前流程设计和代码实现之间最明显的断层之一。

## 5. 组件结构与架构特点

### 5.1 已经形成的结构优点

项目已经初步分成了：

- `screens/` 页面级组件
- `components/` 复用 UI 布局组件
- `contexts/` 全局上下文
- `lib/` 工具函数

其中比较有价值的抽象有：

- `OnboardingLayout`：统一了承接注册、资料录入、偏好录入三步流程的视觉骨架
- `Sidebar`：把主功能区导航集中起来
- `SidebarContext`：给侧边栏开关做了上下文管理

这些说明前任并不是只堆页面，而是已经开始做一定的组件化整理。

### 5.2 还没有收好的结构问题

虽然有结构意识，但目前仍然偏“半成品”：

- `MainLayout.tsx` 已经存在，却没有接入实际路由
- 需要侧边栏的页面都在各自 screen 里手动引入 `<Sidebar />`
- 结果是布局职责分散，页面间容易产生不一致

更具体地说：

- 路由现在统一挂在 `Layout` 下
- 但 `Layout` 只负责一个最外层容器
- 真正的“带侧边栏主应用布局”并没有被路由体系接管

这会导致后面一旦要改主应用布局，需要同时改多个 screen。

## 6. 数据流与状态管理现状

当前项目的状态管理几乎全部是本地 `useState`。

现状如下：

- 表单数据没有统一模型
- 用户资料没有全局 store
- 页面间传递主要靠硬编码和跳路由
- 没有服务端持久化
- 没有 localStorage/sessionStorage 持久化
- 没有 React Query / Zustand / Redux 之类的数据层

这意味着当前前端更像“页面演示器”，而不是“有真实业务状态的应用”。

举例来说：

- 注册页输入邮箱，不保存
- Profile Input 页上传简历，只是 UI 演示，没有真实上传
- Preferences 页的输入结果也没有真正进入用户画像
- Profile Confirmation 页展示的是固定假数据，不是前面三步录入的结果
- Dashboard 和 Chat 使用的是另一套单独维护的 mock 职位数据

## 7. AI 能力接入现状

### 7.1 已接入的部分

`ChatScreen` 里已经直接调用了 Gemini：

- 用户发送消息时会创建 `GoogleGenAI`
- 再调用 `ai.models.generateContent`
- 只把纯文本消息送进模型上下文

这部分说明项目已经尝试把“AI 对话”从概念推进到可交互状态。

### 7.2 未真正接入的部分

真正的职位推荐逻辑并没有接入模型或后端服务，仍然是纯 mock：

- `ChatScreen` 的职位卡片来自 `MOCK_JOBS`
- `DashboardScreen` 的职位列表来自 `appliedJobs` / `suggestedJobs`
- 匹配理由、匹配度、发布时间都是写死的

所以目前“AI 职位推荐”更准确地说是：

- AI 只负责通用聊天文本生成
- 推荐岗位、画像总结、匹配理由仍然主要是前端静态演示数据

## 8. 现阶段的优点

### 8.1 产品表达很清楚

只看前端就能理解产品要做什么：

- 面向香港求职场景
- 面向学生 / 应届生 /早期职业用户
- 卖点是“AI 陪聊 + 画像总结 + 岗位匹配 + 求职管理”

这对交接项目很重要，因为很多继承项目最大的问题不是代码差，而是“看不懂它到底要做什么”。这个项目没有这个问题。

### 8.2 UI 基础不错

这个前端在视觉上已经有比较统一的系统：

- 主色稳定
- 卡片、圆角、边框语言一致
- onboarding、聊天、职位详情、设置页风格统一
- 基础动效让产品观感比普通脚手架强很多

如果后续继续做，不需要从 0 重做视觉基线。

### 8.3 页面覆盖面足够支撑下一阶段开发

关键页面都已经具备雏形：

- onboarding
- chat
- profile
- dashboard
- settings

所以后续工作重点应当是“把壳子接上真实数据和真实逻辑”，而不是从白板重新设计页面体系。

## 9. 主要问题与风险

下面是我认为最关键的几个问题，按影响程度排序。

### 9.1 高风险：Gemini API Key 暴露在前端构建产物中

当前 Vite 配置直接把 `GEMINI_API_KEY` 注入前端：

- `vite.config.ts` 中通过 `define` 注入 `process.env.GEMINI_API_KEY`
- `ChatScreen` 里直接在浏览器端初始化 `GoogleGenAI`

这在原型阶段可以跑通，但如果进入真实环境，会带来明显风险：

- API Key 容易暴露在浏览器端
- 无法做服务端限流、审计、风控
- 也不方便注入系统 prompt、工具调用、上下文拼装等服务端逻辑

建议后续一定改成：

- 前端请求自有后端接口
- 后端代调 Gemini 或其他模型服务

### 9.2 高优先级功能 bug：ProfileConfirmation 编辑数组字段会破坏数据结构

`ProfileConfirmationScreen` 里：

- `experiences[].bullets` 原本是数组
- `extracurriculars[].bullets` 原本也是数组
- 但更新函数 `updateExperience` / `updateExtracurricular` 把任意字段都按 `string` 写回

而编辑 UI 又在用：

- `exp.bullets.join('\n')`
- `ext.bullets.join('\n')`

这会导致一个问题：

- 用户一旦编辑这些 textarea
- `bullets` 就会从数组变成字符串
- 下一次渲染时再调用 `.join()`，很可能直接报错

这个问题不是代码洁癖层面的问题，而是会影响页面正常使用的真实 bug。

### 9.3 业务流程断层：onboarding 完成后并不会进入画像确认分支

`ChatScreen` 里存在依赖 `?onboarding=complete` 的逻辑，但实际导航从 `PreferencesScreen` 直接去 `/chat`。

结果就是：

- 设计上像是先总结画像再推荐岗位
- 实际上用户一进聊天就直接看到推荐岗位
- 画像确认这块几乎变成了一段“写了但走不到”的逻辑

这说明前端当前还没有真正完成从 onboarding 到 AI matching 的流程收口。

### 9.4 架构问题：主布局未统一，Sidebar 接入方式分散

虽然项目有 `MainLayout.tsx`，但并没有用到。

当前需要侧边栏的页面全部自己渲染 `Sidebar`，会带来几个后果：

- 布局逻辑重复
- 页面之间更容易出现细微差异
- 移动端适配难统一
- 后续加 route guard / header / breadcrumb 时不够顺手

这是一个典型的“原型能跑，但接手后越改越散”的信号。

### 9.5 移动端支持不完整

侧边栏本身在小屏下是隐藏的，但移动端导航没有真正补齐：

- `Sidebar` 在 `md` 以下直接不显示
- `ChatScreen` 虽然有 mobile header 和菜单按钮
- 这个菜单按钮没有绑定任何逻辑
- 其他带 sidebar 的页面也没有可用的移动端导航替代

所以这个项目目前并不能说真正支持 mobile，只能说“部分页面做了移动端视觉适配”。

### 9.6 所有核心业务数据都还是 mock

最核心的数据都没有进入真实数据流：

- 用户画像是固定假数据
- 岗位推荐是固定假数据
- 已投递岗位是固定假数据
- Apply / Not interested 只是前端提示或本地追加消息

这意味着当前项目虽然“像一个求职产品”，但还没有进入“产品工程化”的门槛。

### 9.7 表单层还停留在视觉完成阶段

例如：

- 输入框大多没有受控状态
- 没有表单校验
- 没有字段错误提示
- 没有必填项拦截
- 没有上传进度与失败处理
- 没有脏数据保护

所以现阶段这些页面更像“信息采集界面 demo”，不是可以稳定承载真实用户输入的表单系统。

### 9.8 README 和工程文档不足

当前 `README.md` 只回答了怎么跑起来，没有回答：

- 项目是什么
- 页面流程是什么
- 数据从哪里来
- 哪些是 mock，哪些是真实能力
- 部署方式是什么
- 后续开发应该从哪里接手

对继承项目来说，文档不足会直接提高接手成本。

## 10. 工程成熟度判断

如果按阶段来给这个项目前端打标签，我会这样定义：

- 视觉成熟度：中高
- 页面完成度：高
- 业务完成度：低
- 工程化成熟度：中低
- 可上线程度：低

换句话说：

它已经不是“空白脚手架”，但也还远没到“可直接接业务上线”的阶段。

## 11. 建议的接手路线

如果要继续做这个项目，我建议不要一上来就大改 UI，而是按下面顺序推进。

### 第一阶段：先补齐底层安全和运行稳定性

优先做这些事：

1. 把 Gemini 调用从前端移到后端
2. 修掉 `ProfileConfirmation` 里数组字段编辑的运行时 bug
3. 定义统一的用户画像和职位数据类型
4. 把 onboarding 数据真正串到 profile / chat / dashboard
5. 把 `MainLayout` 接回路由，统一主应用布局

这一阶段的目标不是“功能变多”，而是让项目从 demo 进入“可以持续迭代”的状态。

### 第二阶段：把 mock 换成真实业务链路

建议逐步落地：

1. 简历上传接口
2. 简历解析结果结构化存储
3. 用户偏好入库
4. 职位源接入
5. 推荐结果生成与回传
6. Apply / Pass 行为持久化

做到这里，产品才会第一次具备真实可用性。

### 第三阶段：再做体验和增长能力

在链路跑通后，再补：

1. 移动端完整导航
2. 多语言策略
3. 空状态、错误状态、loading 状态
4. 埋点分析
5. 权限与登录态
6. 设置页真实功能

## 12. 我对这个项目的最终判断

如果站在“继承项目”的视角，我会把它定义为：

一个方向正确、界面不错、产品表达清晰，但业务和工程都还没有真正收口的 AI 求职助手前端原型。

它最值得保留的是：

- 产品路径
- UI 资产
- 页面骨架
- AI 求职助手这个交互框架

它最需要尽快补的不是视觉，而是：

- 真实数据流
- 安全的 AI 接入方式
- 页面间状态打通
- 主布局收敛
- 表单与资料编辑的可靠性

## 13. 本次查看时的补充说明

本次我没有跑通完整构建验证，原因是当前目录下缺少 `node_modules`。尝试执行 `npm run lint` 时，命令实际失败在 `tsc: command not found`，说明依赖尚未安装，因此本分析主要基于源码静态阅读完成。

这不会影响对项目结构和实现状态的判断，但会影响对“当前代码是否完全可编译”的最终确认。
